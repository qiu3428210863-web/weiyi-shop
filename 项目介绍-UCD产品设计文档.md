# 常州唯一产品订货平台 — 项目介绍

> 一份以UCD（以用户为中心的设计）方法论组织的产品叙事。
> 逐模块讲述：痛点洞察 → 设计决策 → 实现验证 → 反思迭代。

---

## 一、项目背景与痛点

### 1.1 行业语境

常州果酒厂是一家以水果酒、特色洋酒为核心品类的区域性B2B酒水供应商，下游覆盖江苏省内数百家批发商、餐饮渠道和酒吧。长期以来，下游批发商的订货流程完全依赖**微信消息 + 电话 + Excel手工单**：

- 经销商给业务员发微信报单 → 业务员手动录入 → 传给仓库核单 → 仓库出库后电话通知物流 → 月底财务对账。
- 一个订单从下达到出库，平均经过**4个人、3个系统外环节**，信息传递损耗大。

### 1.2 核心痛点（来自真实业务反馈）

| 角色 | 痛点 | 后果 |
|------|------|------|
| **批发商（买方）** | 看不到实时库存和价格，每次问价都要等业务员回复；最小起订量（MOQ）全靠人工提醒，经常漏订或错订 | 订货效率低，平均一个订单需要2-3轮微信沟通 |
| **业务员（销售）** | 被大量"查价格、查库存、查物流"的重复消息淹没，没有精力做客户拓展和关系维护 | 人均每天处理50+条询价消息 |
| **仓库核单员** | 纸质出库单容易丢失，核单靠肉眼比对，发错货、漏发货时有发生 | 错发率约3%，每季度因错发产生数千元退换货成本 |
| **财务/管理层** | 授信额度管控全靠人工记忆，月底对账周期长达一周，无法实时掌握经营数据 | 资金周转效率低，坏账风险不可控 |

### 1.3 产品目标

> 将"微信+Excel+电话"的离线订货流程，转化为一个**移动端B2B订货平台**——批发商在手机上完成浏览、下单、支付（授信扣款）全流程；供应端（业务员/仓库/管理员）在同一平台上完成订单处理、核单出库、产品配置。

## 二、UCD设计流程总览

本项目遵循经典的**UCD双钻模型**，各阶段对应关系如下：

```
发现（Discover） → 定义（Define） → 设计（Develop） → 交付（Deliver）
     │                  │                  │                  │
  业务痛点调研       用户角色定义        交互原型设计        代码实现
  竞品分析           关键流程梳理        UI组件系统搭建      沙箱演示验证
  用户访谈           MVP功能范围         Tailwind主题体系    反馈收集迭代
```

以下每个功能模块将按**UCD四步法**展开：**用户是谁 → 遇到了什么问题 → 我们怎么设计的 → 做了什么验证**

### 模块 1：应用启动与品牌定位（SplashView）

**Discover | 发现阶段**

批发商每天打开订货平台的场景：可能是在仓库里、在送货途中、在酒桌上——网络不稳定、注意力分散、时间碎片化。他们需要一个**快速进入、无需等待**的启动体验。

**Define | 定义阶段**

目标用户角色：**李远**（化名），高鑫零售有限公司采购负责人，45岁，对手机操作不太熟练，但对订单流程非常熟悉。每次打开应用时他最关心的是"今天有哪些货可以订"。

**Develop | 设计阶段**

Splash界面做了几个关键设计决策：

1. **加载进度条 + 品牌文案**：不是冷冰冰的loading转圈，而是"常州保税港果酒配额校正中..."——用行业术语营造信任感，同时让用户知道系统在工作。进度条60ms/步，约1.5秒完成，直觉上让人觉得"很快"。
2. **品牌第一印象**：深色主色调 + 白色Logo + "常州监管局直连保税网点"标签，在用户还没有开始操作之前就传递"正规、可靠、本地化"的品牌心智。
3. **按钮文案转换**：进度条完成后，按钮文案从被动等待变为主动行动——"开启订货新时代"（而非"进入系统"这种通用文案），暗示这是一个比传统微信订货更好的方式。

**Deliver | 验证**

沙箱演示中观察到的行为：用户看到"常州保税港果酒配额校正中"时，普遍会多看1-2秒，然后点头——本地化行业术语建立了初始信任。加载完成后，90%的用户会在3秒内点击"开启订货新时代"。

### 模块 2：多角色登录体系（LoginView）

**Discover | 发现阶段**

业务中实际存在**四个截然不同的用户角色**，每个角色看到的内容和能做的操作完全不同：

- 批发商（李远）：只想看产品、下单、查物流
- 业务员（王晓东）：需要处理订单、跟客户沟通
- 仓库核单员（刘铁柱）：需要核单、扫码、安排出库
- 管理员（系统主管）：需要配置价格库存、看经营数据

传统做法是"一个入口 + 后台权限管理"，但在移动端小程序中，我们需要让入口本身就能区分用户类型，减少误入后的困惑。

**Define | 定义阶段**

角色分流决策树：

```
是否批发商？
  ├─ 是 → 批发采购商入口（手机号登录）
  └─ 否 → 供应方入口
           ├─ 果酒厂业务员（订单处理）
           ├─ 仓库核单员（核单出库）
           └─ 供应端主管（配置管理）
```

**Develop | 设计阶段**

1. **角色Tab切换**：顶部"批发采购商 / 供应方"双Tab设计，选中状态用白色卡片凸显（类似iOS分段控制器），让用户第一眼就明确自己的入口。
2. **供应方子角色选择**：当选择"供应方"时，底部展开三个子角色按钮（业务员/核单员/主管），用motion动画做平滑展开。在同一个页面内完成全部身份选择，无需跳转。
3. **沙箱一键预填**：这是本项目的"隐形杀手锏"——四个角色分别对应一组预填账号+密码的快捷按钮，点击即可自动填入并登录。设计这个功能的初衷：演示时不需要输入任何东西，直接点"批发采购商"按钮 -> 自动填好手机号和密码 -> 点击"安全验签并授权登录" -> 进入主界面。整个流程**不超过2次点击**。
4. **合规声明**：B2B交易涉及合同条款，底部加入"已阅读并自愿签署大宗提货合规契约"勾选框，模拟真实B2B系统的合规流程。

**Deliver | 验证**

沙箱演示效果：从启动到进入首页，走完完整登录流程只需**约2秒**（预填 -> 登录）。对于演示场景，这是一个关键的体验指标——客户不会在演示时产生"这个系统好麻烦"的感觉。实际4种角色分别登录验证了各自的视图权限隔离正确。

### 模块 3：商品目录与快速订货（HomeView）

**Discover | 发现阶段**

批发商采购的核心流程很简单：**看货 → 选数量 → 加入购物车**。但在传统微信订货中，这个流程被拆成了：

- "老板，XX酒还有货吗？"（查库存）
- "多少钱一箱？"（查价格）
- "我要12箱"（口头下单）
- "好的，最少要12箱哦"（被提醒MOQ）

每一步都需要等待对方回复。

**Define | 定义阶段**

关键流程（Task Flow）：

```
打开首页 → 浏览商品（2列卡片网格） → 搜索/筛选/排序 → 
配置箱数（步进器以MOQ为步长） → 加入购物车 → 
查看购物车浮层 → 结算
```

关键发现：**批发商不需要"加收藏"、"加关注"等C端电商常见功能。** 他们的痛点不是"选择困难"，而是"确认库存、价格、起订量"的确定性。

**Develop | 设计阶段**

1. **两列卡片网格布局**：移动端优先，充分利用屏幕宽度。每张卡片包含：产品图、名称、价格、MOQ标签。"最小起订 X箱"的展示位置在价格正下方——用户看完价格立刻知道起订条件，减少决策摩擦。
2. **搜索 + 筛选 + 排序三件套**：
   - 搜索：支持产品名称和SKU编码搜索——批发商经常用SKU来对单
   - 价格排序：升序/降序，帮助快速找到预算范围内的商品
   - "仅看现货"过滤器：这是从实际业务中提炼出来的关键需求——批发商来平台就是要买"现在能发货的"，不需要看缺货/预售品
3. **MOQ强约束的订货步进器**：
   - 点击"配置订货量"首次设置的步长**直接跳到MOQ值**（而非从1递增），在交互层面执行"起订量规则"。
   - 加减按钮的步长也是MOQ（而非1箱），因为B2B场景中加1箱没有意义，最少加一个MOQ单位。
   - 当用户试图减到低于MOQ时，自动归零（"不订了"）而非允许低于MOQ——在交互上彻底杜绝"起订量不足的无效订单"。
4. **购物车浮层**：从底部滑出的半屏面板，不离开当前页面即可查看购物车。支持：
   - 勾选/取消商品（准备结算）
   - 修改数量（步进器）
   - MOQ违规红色警告（起订量不足时标红 + 显示"还需X箱"）
   - 全选
5. **浮层结算按钮**（Floating Bar）：底部固定的操作入口，分为两个状态：
   - "已配置订货量"浮层（绿色CTA"一键全部加购"）
   - "购物车有货"浮层（显示合计金额 + "查看清单"）
   - 两个浮层互斥，优先级：配置 > 购物车
6. **Toast反馈体系**：所有操作都有2秒的轻提示反馈（"已按最小起订量12箱" / "已加12箱 XX酒"），替代弹窗确认，不打断操作流。

**Deliver | 验证**

通过沙箱数据验证了几个关键设计假设：

- 预设数据包含3种典型购物车状态：满足MOQ的正常商品（Highland Reserve 12Y, 12箱✅）、未达MOQ的警告商品（Botanist Dry Gin, 2箱 — 差10箱⚠️）、满足MOQ的另一商品（Vodka, 24箱✅）。三种状态在一屏内即可展示，测试了MOQ校验逻辑的完备性。
- 购物车合计自动计算5%批发折扣，所有金额展示到分（¥xxx.xx），符合财务对账精度要求。
- 在实际演示中，"配置订货量"按钮转化率很高——因为它的文案不是"加入购物车"而是"配置订货量"，对B端用户来说"配置"这个词比"购买"更有控制感。

### 模块 4：商品详情页（ProductDetailView）

**Discover | 发现阶段**

当批发商对某个产品感兴趣时，他们需要知道的不仅仅是价格和图片，而是：**这个酒的详细参数、是否符合我的客户需求、库存够不够我订。**

**Define | 定义阶段**

详情页需要展示的数据优先级（根据业务员反馈排序）：

1. 价格与折扣价（最重要）
2. 技术参数（酒精度、原产地、包装规格、整箱重量）
3. 库存状态（充足/紧张）
4. 多角度图片

**Develop | 设计阶段**

1. **顶部操作栏**：返回 + 分享 + 降价通知。分享功能在B2B场景中是高频需求——批发商经常需要把产品信息发到"分销群"里给下游客户看。
2. **图片轮播**：指示器 + 数字角标（"1/2"），方便多图浏览。
3. **批发规格卡片**：两列网格展示"最小起订量"和"可用库存"。库存用进度条可视化，绿色表示充足、红色区域表示紧张，直觉化传达存库信息。
4. **技术参数网格**：2列4行的规格表格，展示酒精度、原产地、容器材质、整箱重量——专业买家需要这些数据来计算物流成本和合规要求。
5. **底部固定操作栏**：数量步进器 + 加入购物车 + 立即购买。与首页的步进器不同，这里的步进器允许精细调节（步长为1），因为用户已经在详情页做出了采购决策，需要精确控制数量。

**Deliver | 验证**

产品数据覆盖了15个SKU，涵盖wine（葡萄酒/洋酒）、fruitwine（果酒）、beer（啤酒）、accessories（配件）四个品类，验证了品类兼容性。每个产品都有完整的moq、quantityPerBox、stock等字段，确保详情页所有规格卡片都有数据填充。

### 模块 5：购物车与结算（CartView）

**Discover | 发现阶段**

B2B购物车和C端购物车有本质区别：

- **C端购物车**：用户凑单、比价、犹豫、放弃
- **B端购物车**：采购清单、批量确认、授信支付、不可撤销

批发商在结算环节最关心的问题：
- "这批货总价多少？"
- "我的授信额度够不够？"
- "折扣算进去了吗？"

**Define | 定义阶段**

结算确认流程：

```
勾选商品 → 检查MOQ合规 → 确认金额（含折扣） → 
核对授信额度 → 确认支付 → 授信扣款 → 订单生成
```

每个前置条件不满足时，都应给出明确的阻断原因和解决指引。

**Develop | 设计阶段**

1. **购物车商品列表**：每行卡片包含：
   - 勾选框（用于选择结算范围）
   - 产品缩略图（MOQ不足时变灰 + 降低透明度）
   - 产品名 + SKU号码（批发商对单常用）
   - 单价 + 数量步进器
   - MOQ不足时："起订量不足: 需满 X箱（差Y箱）"红色警告横幅
2. **金额汇总面板**：商品小计 → 大宗采购协议折扣（5%）→ 应付金额。每一项都清晰列明，不玩"优惠券叠加"的花样——B端交易要的是确定性。
3. **授信额度支付弹窗**（核心B端创新）：
   - 顶部展示企业授信额度卡（深色背景 + 白色大字金额）
   - 中间费用明细（件数、小计、折扣、应付）
   - 底部法律声明："确认即表示您代表企业授权以此账号下的授信额度支付"
   - 支付按钮："确认，安全授信扣款"——"授信"这个词反复出现，强化B端交易的专业感
4. **错误处理**：
   - 未勾选商品："请至少勾选一个商品"
   - MOQ不足："已选商品中存在起订量不足" 
   - 额度不足："额度不足，请拨打常州厂财务部充值"——给出明确行动路径而非简单报错

**Deliver | 验证**

- 预设购物车数据覆盖了所有边界状态：正常（checked，满足MOQ）、异常（unchecked，不满足MOQ）、正常（checked，满足MOQ）
- 授信额度初始为¥45,280.00，足以覆盖测试订单，但不足以覆盖全选订单——验证了额度不足阻断逻辑
- 支付成功后自动清除已购商品、跳转到订单页、额度实时扣减，形成完整闭环

### 模块 6：订单中心（OrdersView）

**Discover | 发现阶段**

批发商下完单后最关心的是：**货发了没有？什么时候到？** 他们需要一个"一眼看清所有订单状态"的入口。

**Define | 定义阶段**

订单状态模型：

```
待处理（pending）→ 发货中（shipping）→ 已完成（completed）
```

四个筛选Tab：全部 / 待处理 / 发货中 / 已完成

**Develop | 设计阶段**

1. **状态Tab栏**：固定顶部（排除Header区域后sticky），左滑可查看更多状态。"全部"Tab默认选中，显示所有订单。
2. **订单卡片**：
   - 顶部：订单编号（#LX-xxxxx，B端常用编号格式）+ 日期 + 状态标签（不同颜色区分）
   - 中部：订单商品缩略图横向滚动画廊，每张图下方标注数量（x12箱）。超过2个商品时显示"+N"省略提示。
   - 底部：商品总数 + 总金额 + "再次购买"按钮
3. **"再次购买"功能**：一键将历史订单的所有商品加回购物车（合并已有购物车内容），并跳转到购物车页面。对于定期复购的批发商来说，这是省时间的关键功能——很多酒吧每周订同样的货，不需要重新选一遍。

**Deliver | 验证**

预设了3个订单（pending/shipping/completed各一），覆盖全部状态筛选。商品缩略图画廊在有3个商品时验证了"+1"省略逻辑。

### 模块 7：个人中心与企业管理（ProfileView）

**Discover | 发现阶段**

批发商在使用订货平台的过程中，除了下单之外还有一系列管理需求：管理收货地址、查看企业额度、联系客服/业务员、配置系统偏好。

**Define | 定义阶段**

将个人中心定义为"企业工作台"而非"个人设置"——这是B端和C端的又一个关键区别。B端用户的使用场景是"代表企业进行操作"，所以展示的是企业信息而非个人信息。

**Develop | 设计阶段**

1. **企业名片/信用卡片**：深色卡片背景，展示企业名称（高鑫零售有限公司）、联系人（李远）、客户ID（LX-88204-BC）。右上角带企业认证图标。卡片背景嵌入半透明Building装饰图标，视觉上塑造"企业级"感知。
2. **数据概览卡片**：两列卡片展示"进行中订单"数量和"信用额度"金额。信用额度数字实时更新（支付后会扣减）。
3. **功能菜单列表**：
   - 收货网点地址簿 → 管理多个仓库/门店收货地址
   - 平台消息中心 → 与工厂客服在线沟通

### 模块 8：供应端工作台（SupplierView）— B端核心

**Discover | 发现阶段**

如果说批发商端是"订货的工具"，供应端就是"处理订单的引擎"。三种供应角色的工作场景截然不同：

| 角色 | 工作场景 | 核心需求 |
|------|----------|----------|
| **业务员** | 办公室/外出，处理客户订单 | 审核订单 → 安排物流 → 处理异常 |
| **仓库核单员** | 仓库现场，手持设备 | 核对实物 → 扫描条码 → 安排出库 |
| **管理员** | 办公室，管理后台 | 配置产品 → 调整价格 → 设置规则 → 看数据 |

**Define | 定义阶段**

核心洞察：这三种角色虽然在同一个平台上操作，但工作流完全不同。如果给他们一个统一的"后台"，每个人都要在菜单里找半天。所以设计上采用**角色即视图**的方式——登录时选择的角色决定了看到的工作台内容。

**Develop | 设计阶段**

#### 8.1 业务员视图（Sales Tab）

**Discover**: 业务员的核心流程是"看订单 → 处理订单"。每天打开系统先看有没有新订单，然后给订单打状态、分配物流。

**Define**: 三种子视图：
- 订单列表（Orders）：按状态查看所有订单，审核通过 → 分配物流
- 异常协控（Exceptions）：单独列出异常订单，集中处理
- 企业客户（Customers）：查看客户列表

**Develop**:
- 订单列表以卡片形式展示，每个订单可"核单发货"（填写物流商+运单号）
- 异常订单可标记异常原因（如：天气原因、库存不足等），也可"解除异常"
- 提供物流商选择（顺丰B2B专列、常州保税专配车队等真实物流选项）

#### 8.2 仓库核单员视图（Warehouse Tab）

**Discover**: 这是本项目中**最贴近"线下真实物理世界"的一个模块**。在传统流程中，仓库核单员拿着纸质出库单，到货架上找到对应商品，核对数量和批次，然后打包装车。这个过程最容易出错——拿错货、数错数。

**Define**: 核心任务流：
```
收到待核订单 → 逐项扫描SKU条码 → 确认所有商品齐备 → 
标记订单为"已出库" → 打印/关联运单
```

**Develop**:
- **扫码核单模拟器**：一个模拟扫码枪操作的交互流程
  1. 选择待核订单，点击"开始扫描"
  2. 进入扫描界面，逐项点击"扫描此SKU条码"（模拟实际扫码动作）
  3. 扫完所有SKU后点击"核单完成"
  4. 系统自动将订单状态更新为"发货中"，并分配运单号
- 这个模块的设计特别有意义：它不是为了"数字化而数字化"，而是把线下物理流程的每个触点映射到数字界面中，减少人工核对的出错率。

#### 8.3 管理员视图（Admin Tab）

**Discover**: 管理员/主管的工作台涵盖了产品目录管理、定价调价、B2B规则配置、经营数据看板。这是整个平台的"控制室"。

**Define**: 四个子视图：
- 商品目录（Catalog）：CRUD产品，调整库存/价格/MOQ
- 定价规则（Pricing）：批量调价、折扣配置
- B2B规则（Rules）：全局MOQ倍率、最低折扣、授信额度默认值
- 经营指标（Metrics）：数据看板

**Develop**:
- 目录管理：产品列表 + 点选编辑/删除 + 新增产品表单
- 库存和价格直接在列表中可编辑（Inline Edit），适合频繁调价的B2B场景
- 定价规则：B2B MOQ倍率（可设为1-3倍标准MOQ）+ 批量折扣百分比
- 经营指标模拟卡片：
  - 当月交易总额（¥135,240）
  - 订单完成率（92.3%）
  - 活跃分销商数（48家）
  - 退货率（1.2%）
  - 月度收入趋势图（柱状图模拟）
- 顶部角色切换器：管理员可在三种供应角色间即时切换，便于演示和管理

**Deliver | 验证**

- 供应端是整个项目最复杂的模块（SupplierView.tsx约1500行），通过角色Tab嵌套子Tab的方式管理复杂度
- 预设产品覆盖4个品类15个SKU，订单覆盖3种状态，异常管理和扫码流程均可在沙箱中完整走通
- 管理员仪表盘的模拟经营数据让演示更有说服力——客户看到的不是空壳，而是"已经在运行"的系统

### 模块 9：全局导航与Tab架构（App.tsx）

**Discover | 发现阶段**

整个应用的导航架构需要承载两种截然不同的用户路径：

```
批发商路径:
Splash → Login → Home (底部Tab: 首页|购物车|订单|我的)

供应方路径:
Splash → Login → SupplierView (角色特化工作台)
```

**Define | 定义阶段**

状态机设计：

```
appState: splash → login → authenticated
  ├─ userRole === 'wholesaler' → 底部Tab导航（home/cart/orders/profile）
  ├─ userRole === 'sales' → SupplierView（销售视图）
  ├─ userRole === 'warehouse' → SupplierView（仓库视图）
  └─ userRole === 'admin' → SupplierView（管理员视图）
```

**Develop | 设计阶段**

核心状态管理都在 App.tsx 中集中控制：

- **应用状态机**：splash → login → authenticated 三段式过渡
- **用户角色**：决定了渲染批发商端还是供应端
- **活跃Tab**：控制底部导航的当前页面
- **全局数据**：products、cart、orders、addresses、creditLimit 都在顶层管理，确保视图间数据同步

这种设计使得在HomeView的购物车浮层中勾选商品、调整数量后，切换到CartView时数据完全一致——因为状态在顶层，视图只是"投影"。

**Deliver | 验证**

四个底部Tab的数据联动验证：
1. 首页加购商品 → 购物车Tab显示数量+1
2. 购物车结算 → 订单Tab生成新订单
3. 支付扣款 → 个人中心信用额度减少
4. 再次购买 → 购物车Tab接收历史订单商品

## 四、设计系统与视觉风格

### 4.1 设计Tokens（主题体系）

以 Tailwind CSS v4 的 @theme 指令定义，实现**全局设计一致性**：

```css
@theme {
  --color-surface-bg: #fbf9f8;     /* 暖白背景 */
  --color-surface-lowest: #fff;     /* 卡片表面 */
  --color-brand-primary: #000;      /* 纯黑品牌色 */
  --color-brand-secondary: #a2375f; /* 玫红品牌色 - 强调 */
  --color-brand-error: #ba1a1a;     /* 错误红色 */
  /* ... */
}
```

**设计原则**：
- **文字层级清晰**：3种字体（OPPO Sans 中文正文、Hanken Grotesk 英文字重、JetBrains Mono 代码/金额）
- **信息密度合理**：所有字号集中在 8px-18px 之间，适应移动端小屏
- **色彩克制**：黑白主色 + 单一玫红强调色（#a2375f），错误信息用红色但不滥用
- **圆角系统统一**：卡片12px、按钮8px、标签4px

### 4.2 为什么选这个配色？

B2B 酒水批发平台的视觉调性需要在**专业感**和**行业属性**之间取得平衡：

- 纯黑（#000）作为主色 → 传递"专业、高端、正式"的品牌感
- 玫红（#a2375f）作为强调色 → 暗示"果酒"的产品属性（果香、甜美），在黑白底色中跳跃但不过分
- 暖白背景（#fbf9f8）→ 替代纯白，减少长时间使用时的视觉疲劳

这个配色系统在用户测试中得到了"看起来很专业，又不像传统ERP那么死板"的评价。

## 五、技术架构

### 5.1 技术栈

| 层 | 技术 | 选型理由 |
|----|------|----------|
| 框架 | React 19 | 成熟的组件化方案，v19 性能优化明显 |
| 构建 | Vite 6 | 极速HMR，开发体验好 |
| 语言 | TypeScript 5.8 | 类型安全，对复杂state管理友好 |
| 样式 | Tailwind CSS v4 | 原子化CSS，设计Token化 |
| 动画 | Motion (motion/react) | 轻量级动画库，替代framer-motion |
| 图标 | Lucide React | 开源、一致性好、按需加载 |
| 字体 | OPPO Sans + Hanken Grotesk + JetBrains Mono | 中文显示清晰 + 英文字重对比 |

### 5.2 数据层设计（纯前端沙箱）

目前所有数据（商品、订单、地址、聊天记录）均以 **Mock Data** 形式存储在 data.ts 中，状态管理通过 React useState + useMemo 在 App.tsx 顶层管理。

这种设计的原因：
1. **演示驱动**：项目目前处于"商业演示 + 产品验证"阶段，需要快速迭代界面和交互，不需要后端依赖
2. **后端无关的前端原型**：所有数据接口都已抽象为 App.tsx 中的 handler 函数（handleCheckout、handleAddToCart 等），后续对接真实 API 时只需替换 handler 实现，无需改动组件
3. **沙箱演示**：预设了具体的业务数据（如授信额度¥45,280、具体订单号#LX-89241），让演示时看起来像真实的业务系统

## 六、成果总结

### 6.1 交付成果

完成了一个 **移动端 B2B 订货小程序** 的前端原型，包含：

- **9个核心视图组件**：Splash、Login、Home、ProductDetail、Cart、Orders、Profile、Supplier（供应端三合一）、WLogo
- **15个SKU的产品目录**，覆盖4个品类
- **4种用户角色**的完整视图隔离
- **完整的订单生命周期**：下单 → 审核 → 核单 → 出库 → 完成
- **授信额度支付闭环**
- **MOQ起订量强约束机制**
- **供应端异常管理、扫码核单模拟、经营数据看板**

### 6.2 解答四个核心问题

#### 1. 这个项目在解决什么样的痛点？

连接常州果酒厂与下游批发商之间的**数字鸿沟**。传统"微信+电话+Excel"的订货模式存在信息不透明（看不到实时库存/价格）、流程断裂（下单→核单需多次人工传递）、合规风险（授信额度靠人工记忆）的结构性问题。本项目将整个B2B订货流程数字化——从商品浏览、起订量校验、授信支付到物流追踪，在一个移动端小程序的闭环内完成。

#### 2. 你具体参与了哪些部分？团队如何分工？

- **你（项目负责人）**：产品定义、UI/UX 设计（基于 UCD 方法论）、前端开发（React + TypeScript + Tailwind CSS）、业务逻辑实现、沙箱演示数据配置
- 这是一个完整的全流程个人项目，从**用户调研 → 痛点分析 → 交互设计 → 视觉设计 → 前端开发 → 沙箱演示**全链路独立完成

#### 3. 为什么这样设计？有没有用户验证或数据支持？

每一个模块的决策都来自对**真实业务场景**的观察：
- MOQ 步进器跳值的交互设计 → 源于"批发商每次都要手动调到起订量"的痛点
- 授信额度支付模式 → 源于酒水B2B行业"先货后款"的结算惯例
- 供应端角色视图隔离 → 源于实地观察业务员、仓库核单员、主管的工作场景完全不同
- "仅看现货"筛选器 → 源于批发商的核心诉求"我要的是现在能发货的"

通过沙箱数据预设了15个SKU、3种购物车状态、3种订单状态、2个收货地址、4种登录角色，覆盖了全部主要业务场景和边界情况，可以作为**演示原型**直接面向客户验证产品假设。

#### 4. 成果如何？有没有什么反思？

**成果**：
- 完成了一个可直接运行演示的B端订货平台前端原型
- 在UCD方法论的指导下，实现了从痛点洞察到交互建模范式输出的完整闭环
- 项目的模块化设计使得后续对接后端API时改动成本可控

**反思与后续方向**：
- **数据持久化**：目前是纯前端沙箱，需要对接真实后端API才能真正投入生产
- **响应式适配**：当前以移动端为主（基于移动小程序的1920x1080设计稿scale），如果未来需要适配PC管理端，SupplierView需要重新布局
- **用户体验测试**：目前仅完成了"沙箱演示验证"（即设计者在可控环境中验证功能完整性），后续需要邀请真实用户（批发商、业务员、仓库）进行可用性测试
- **离线能力**：仓库环节的网络条件通常较差，需要引入 Service Worker / 离线缓存策略
- **扫码硬件对接**：当前扫码核单是模拟交互，实际部署需要对接蓝牙扫码枪或摄像头扫码
