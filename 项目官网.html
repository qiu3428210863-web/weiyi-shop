Discover · 发现阶段

传统订货流程的四大痛点

在常州果酒厂的真实业务中，四个角色的日常协作完全依赖于微信消息、电话和 Excel 手工单，信息传递损耗严重。

🏪

批发商（买方）

看不到实时库存和价格，每次问价都要等业务员回复；最小起订量全靠人工提醒，经常漏订或错订。

平均一个订单需要 2-3 轮微信沟通
👤

业务员（销售）

被大量"查价格、查库存、查物流"的重复消息淹没，没有精力做客户拓展和关系维护。

人均每天处理 50+ 条询价消息
📦

仓库核单员

纸质出库单容易丢失，核单靠肉眼比对，发错货、漏发货时有发生。

错发率约 3%，每季度产生数千元退换货成本
📊

财务 / 管理层

授信额度管控全靠人工记忆，月底对账周期长达一周，无法实时掌握经营数据。

资金周转效率低，坏账风险不可控
❌ 传统流程 客户微信报单 → 业务员手动录入 → 仓库核单 → 通知物流 → Excel 对账
→
✅ 数字化流程 批发商 APP 下单 → 自动校验 MOQ/额度 → 仓库扫码核单 → 授信扣款 → 实时数据看板

将 4 人 x 3 环节的信息接力，压缩为一人一端的自助闭环

Develop · 设计阶段 — 精选模块

关键功能模块的设计决策

以下四个模块集中体现了本项目的设计理念。每个模块按"业务场景 → 原始诉求 → 设计决策 → 业务规则"的结构展开，追溯每一个交互细节背后的真实业务原因。

01

多角色登录体系 — "角色即视图"

业务场景

四类人使用同一系统，但做的事情完全不同：批发商只想看货下单，业务员要处理订单，仓库要核单出库，管理员要配置商品和看数据。

业务员、工厂、管理员，我们厂里这些人肯定要用系统。但批发商是客户，不能让他们看到工厂后台那些复杂的东西。最好登录的时候就分开。
被否定的方案

传统做法：统一登录入口，登录后靠权限控制菜单可见性。
❌ 批发商面对"订单管理、客户管理、系统设置"等后台菜单不知所措。

设计决策

角色即视图，从登录页分流

批发商和供应方从登录页就分道扬镳。批发商看到的是底部四元素导航的商城界面；供应端内部再按角色显示独立工作台。

角色即视图 · 分离登录入口
业务规则
  • 批发采购商 / 供应方双入口分流
  • 供应端三级子角色：业务员 / 仓库核单员 / 管理员
  • 管理员特有角色切换器（可预览各角色视图）
  • 演示模式：4 个一键预填按钮跳过登录输入
  • 合规勾选框："已阅读并自愿签署大宗提货合规契约"
为什么不在一个界面里做角色切换？ 技术上可行，但业务上不需要——一个人通常只属于一个角色。角色切换器对大多数人来说是噪音，只有管理员需要这个功能。
02

商品目录与快速订货 — B 端 vs C 端的本质差异

业务场景

批发商采购的核心流程是"看货 → 选数量 → 加入购物车"。但在传统模式中，每一步都要微信问业务员——有货吗？多少钱？最少买多少？

批发商能在手机上直接看到价格和库存，不用每次都要问我们。还有那个起订量，每次都要人工提醒，有些人老订不够起订量。
B2C 做法 vs B2B 做法

C 端卡片放"月销 10 万+"、"好评率 98%"，B 端卡片放"库存充足"、"¥428/箱"、"最小起订 12 箱"。
B 端采购决策的心理是确定性——不是从众。

设计决策

两列卡片网格 + MOQ 步进器

两列网格每屏展示 6-8 个商品，满足批发商"快速扫一遍"的浏览需求。步进器以 MOQ 为步长，从交互层面杜绝低于起订量的无效输入。

Poka-yoke 防错设计 · MOQ 步进器
业务规则
  • 价格和库存实时可见，无需询价
  • 点击"配置订货量"直接跳到 MOQ 值（如 12 箱）
  • 加减步长为 ±1 箱（非 MOQ 倍率），方便微调
  • 低于 MOQ 时弹出提示"还差 X 箱起订"
  • 库存用进度条可视化：绿色=充足→红色=告急
为什么不支持少于 MOQ 的数量然后在结算时拦截？ 在结算时才告知"起订量不足"是糟糕的体验——用户应该在第一步就被阻止犯错，而不是最后一步才被告知不行。这叫防错设计（Poka-yoke）。
03

购物车与结算 — "点确认就是签合同"

业务场景

B2B 购物车不是"放进去再看看、比价"，而是"采购清单——确认了就要执行"。批发商关注的是总价对不对、额度够不够、流程不可逆。

批发商选好了之后，能看到总价多少、折扣多少、要付多少。然后从我们给他的授信额度里直接扣。点了确认就不能反悔了。
设计决策

授信扣款 + 合同式支付弹窗

支付方式不是微信/支付宝，而是授信额度扣款——模拟真实 B2B 的"先货后款"信用交易模式。支付弹窗设计得"重"：额度卡 + 费用明细 + 法律声明，每一个元素都是为了强化"这是企业行为"的认知。

授信扣款 · 合同式支付 · B 端信任设计
结算校验规则（多级阻断）
  • 未勾选任何商品 → "请至少勾选一个商品"
  • 勾选商品中存在 MOQ 不足 → 标红并阻断
  • 应付金额 > 授信额度 → "额度不足，请拨打财务部充值"
  • 全部通过 → 显示支付弹窗（额度卡 + 明细 + 声明）
  • 确认支付 → 加载 1.2s → 成功动画 1.5s → 跳转订单页
为什么不用微信/支付宝？ 该公司传统惯例是"先货后款"——批发商先提货，月底结款。授信额度就是信用账期的数字化体现。如果用 C 端的即时支付模式，反而违背了用户的已有习惯。
04

供应端工作台 — 不为三角色造统一后台

业务场景

业务员、仓库核单员、管理员——三个人都在工厂工作，但工作流完全不同。传统做法是一个后台菜单栏靠权限控制可见性，但移动端屏幕小，菜单本身就占用大量空间。

我们厂里的人也要用这个系统。但是这三个人看的东西完全不一样，不要搞一个菜单让他们自己找。
设计决策

三个独立的垂直工作台

供应端不是一个"后台"，而是三个独立的工作台。每个工作台只为该角色的核心任务优化，没有多余的入口。

角色即视图 · 移动端优先 · 任务导向
三种工作台
  • 业务员：合同订单簿 / 异常协控单 / 大宗采购商
  • 仓库核单员：待核清单 → 扫码模拟 → 录入车牌 → 一键出库
  • 管理员：商品管理 / 价格微调 / 供销规则 / 经营数据（SVG 柱状图）
为什么不做统一后台 + 权限控制？ 这在 PC 端是标准做法，但在移动端上，菜单占空间、信息密度低、三个角色的工作流完全不同。给他们一个统一的"后台"让他们自己找功能，效率极低。
设计的本质是理解业务

以上四个模块的设计决策有一个共同点：每个交互细节都不是"拍脑袋"做的，而是从真实的 B2B 酒水批发业务场景中推导出来的。理解业务，比理解界面更重要。
