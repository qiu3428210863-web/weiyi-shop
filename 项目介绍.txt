一、常州唯一数字化订货平台应用启动与品牌定位（SplashView）

  业务场景

  批发商打开订货平台的典型场景：在仓库里搬货、在送货车上、在酒桌上陪客户。网络不稳定、注意力分散、时间碎片化。
  传统订货流程（客户通过微信-业务员或内勤记录订单-核对价格和库存-通知工厂出货-发货和物流跟进-Excel对账结算-售后）

  甲方原始诉求

  ▎ "我们厂在常州做了十几年，下游批发商都认我们这块牌子。但以前他们用微信报单，打开微信就是聊天记录，流程很复杂且没什么品牌感。现在
  ▎ 要做自己的平台，第一眼就要让他们觉得——这是属于我们常州唯一果酒厂自己的正规平台，我们的流程很正规、很专业。"

  翻译成需求语言：
  - 订货流程简单便利（方便用户下单及甲方对接订单）
  - 要有品牌感（一打开就知道是"常州唯一果酒厂"）
  - 要显得正规（B端交易需要信任感）

  "进入系统"是功能描述，"开启订货新时代"是价值主张。这个按钮是批发商从"旁观者"变成"使用者"的心理分界线——点击之前他只是在
  看，点击之后他就是这个新订货方式的参与者了。

  业务规则落地

  交互规则：
  1. 第一次进入时启动页进度达到100%前，按钮不可见（必须等待），后续进入网站后直接跳过启动页（防止过多浪费用户时间）
  2. 点击"开启订货新时代"后不可逆进入登录页（无返回）

  二、多角色登录体系（LoginView）

  业务场景

  在实际业务中，有四类人需要用这个系统，他们做的事情完全不同：

  ┌────────────────┬─────────────┬────────────┬────────────────────┐
  │      角色      │  工作地点   │  常用设备  │      核心操作      │
  ├────────────────┼─────────────┼────────────┼────────────────────┤
  │ 批发商（买方） │ 自家店/仓库 │ 自己的手机 │ 看货、下单、查物流 │
  ├────────────────┼─────────────┼────────────┼────────────────────┤
  │ 业务员         │ 办公室/外出 │ 公司手机   │ 处理订单、联系客户 │
  ├────────────────┼─────────────┼────────────┼────────────────────┤
  │ 仓库核单员     │ 仓库现场    │ 手持终端   │ 扫码、核单、出库   │
  ├────────────────┼─────────────┼────────────┼────────────────────┤
  │ 系统管理员         │ 办公室      │ 电脑/手机  │ 配置产品、看数据   │
  └────────────────┴─────────────┴────────────┴────────────────────┘

  传统ERP的做法是：所有人同一个登录入口，登录后在后台分配权限。但在移动端场景中，这种做法的体验很差——批发商登录后看到的
  是"订单管理、客户管理、商品管理、系统设置"这种后台菜单，他根本不知道点哪个。

  甲方原始诉求

  ▎ "业务员、仓库、管理员，我们厂里这些人肯定要用系统。但批发商是客户，不能让他们看到工厂后台那些复杂的东西，他们只需要
  ▎ 看货下单。最好登录的时候就分开。"

  翻译成需求语言：
  - 批发商和工厂内部人员必须从登录页就分流
  - 批发商的界面要极简，能够高效传递信息，"像个商城"而不是"像个后台"
  - 工厂内部三个角色的工作台也要各自独立

  设计决策原因

  为什么做了"批发采购商 / 供应方"双端 + 供应方三级子角色？

  这个设计的核心逻辑是角色即视图——选择什么角色，就看到什么界面，不需要额外的权限配置。

  - 批发商登录 → 底部四元素导航（首页/购物车/订单/我的）
  - 业务员登录 → 订单处理工作台（订单列表/异常协控/客户列表）
  - 仓库核单员登录 → 核单工作台（待核清单/扫码枪界面）
  - 管理员登录 → 配置工作台（商品管理/定价/规则/数据看板）

  四种人有四种完全不同的界面，每个界面只为该角色的核心任务优化，没有多余的入口。

  为什么不在一个界面里做角色切换？

  技术上是可以的，但业务上不行。因为在实际业务中，一个人通常只属于一个角色。批发商不会需要切换到"仓库核单员"视图，业务员
  也不需要切换到"管理员"视图。一个角色切换器放在界面里，对大多数人来说是噪音。只有管理员（超级管理员）有这个需要——所以供
  应端顶部做了一个角色切换选择器，但那是管理员特有的功能。

  为什么做了沙箱快捷预填按钮（4个一键填入角色）？

  这个设计表面上是"演示方便"，但深层原因是：在真实演示场景中，输入手机号+密码这个动作就是最大的体验卡点。
  客户来考察系统，看的是"进系统之后好不好用"，而不是"登录这个动作本身"。如果每次演示都要让客户看着你输入账号密码10秒钟，
  注意力就流失了。

  所以设计了"点击按钮 → 自动填入 → 点击登录 →
  进入系统"的两步流程。这个功能在演示时会被用到，但上线后会隐藏或删除。

  为什么做了合规勾选框（"已阅读并自愿签署大宗提货合规契约"）？

  B2B交易涉及合同条款、法人授权、保税合规——这不是C端买衣服，勾了"同意用户协议"也没人看。在B2B场景中，每一笔订单都是具有
  法律效力的采购合同。登录时的合规勾选框实质上是一种数字签名的前置确认——让批发商在登录时就知道"我在这里下的单是有法律效
  力的"。

  三、商品目录与快速订货（HomeView）

  业务场景

  批发商采购的核心流程：看货 → 选数量 → 加入购物车。看起来和C端电商一样，但细节上完全不同。

  在传统微信订货中，一个典型对话是这样的：

  ▎ 批发商："老板，XX酒还有货吗？"
  ▎ 业务员："有的，现在XX酒批发价428一箱，12箱起订。"
  ▎ 批发商："我要12箱。"
  ▎ 业务员："好的，我给你下单。"

  每一步都需要等待对方回复。一个批发商一天可能要问5-10种酒，平均一个单品需要2-3轮沟通。效率极低。

  甲方原始诉求

  ▎ "批发商能在手机上直接看到价格和库存，不用每次都要问我们。还有那个起订量，每次都要人工提醒，有些人老订不够起订量，我
  ▎ 们也很为难——不发货吧得罪客户，发货吧我们亏运费。"

  翻译成需求语言：
  - 价格和库存对批发商可见，不需要问业务员
  - 最小起订量（MOQ）要在界面上强制校验，不能依赖人工提醒
  - 批发商自己完成选量操作，不需要人工介入

  设计决策原因（找得最深的部分）

  为什么用两列卡片网格而不是列表？

  移动端屏幕宽度有限。列表布局（左边图片右边文字）在手机上每屏只能展示3-4个商品。两列网格每屏可以展示6-8个。批发商浏览商
  品的目标不是"仔细看每个商品的细节"，而是**"快速扫一遍，看看有没有我要的"**——两列网格正好满足这个扫描式浏览的需求。

  为什么卡片上放的是"最小起订X箱"而不是"好评率"或"销量"？

  这是B端和C端的本质区别。C端电商在卡片上放"月销10万+"、"好评率98%"，因为C端消费者的决策心理是"从众"——别人买得多我就买。
  但B端批发商的决策心理是**"确定性"**——我的客户在等这批货，我必须确认我能买到足够的数量。所以卡片上最重要的信息依次是：
  有没有货（stock）→ 多少钱（price）→ 最少买多少（MOQ）。

  为什么步进器的步长是MOQ而不是1？

  这个设计在代码里看起来很特别，但业务逻辑非常扎实：

  - 批发商点击"配置订货量" → 数量直接跳到MOQ值（比如12箱），而不是从1递增。
  - 加减按钮的步长也是MOQ是±1箱，方便用户自由增加起订量。
  - 当用户减少到低于MOQ时会弹出提示提醒用户还需加多少箱才能起订

  为什么不支持"少于MOQ的数量"？因为在实际业务中，少于MOQ的订单工厂不发货。如果界面允许用户选到5箱、8箱，然后在结算时告诉
  他"起订量不足"，这是一个糟糕的体验——用户在第一步就应该被阻止犯错，而不是在最后一步才被告知不行。

  这个设计逻辑叫"防错设计（Poka-yoke）"——在交互层面彻底杜绝无效输入的可能性。

  为什么做了"已配置订货量"和"购物车有货"两个互斥浮层？

  这两个浮层对应批发商的两个独立操作阶段：

  1. 配置阶段（浏览商品时）：看到了感兴趣的商品，在卡片上设置了数量（比如12箱），但还没加购物车。这时候显示绿色浮层"一键
  全部加购"——把当前所有"已配置但未加购"的商品一次性加入购物车。
  2. 购物阶段（加购之后）：商品已经在购物车里了，显示"查看清单"浮层——点击查看购物车，准备结算。

  为什么互斥？因为这两个阶段不会同时发生。如果用户在配置阶段，他关心的就是"把配好的量加到购物车"；如果已经在购物车有商品
  了，他关心的就是"去结算"。同时显示两个浮层只会增加认知负荷。

  为什么做了搜索但没做分类筛选？

  代码里实现了搜索（名称+SKU）、价格排序、仅看现货，但没有"按品类筛选"。为什么？从数据来看：

  - 项目有15个SKU，分4个品类（wine/fruitwine/beer/accessories）
  - 15个商品在移动端两列网格中大约是2.5屏
  - 对于2.5屏的内容量，搜索比分类筛选更高效

  批发商找商品通常有两种方式：
  1. 认品牌/认名字：直接搜"XX酒"或SKU编码
  2. 扫一遍全目录：对于小规模品类（15个），全部浏览并不费劲

  如果未来商品扩展到50+SKU，需要补充品类筛选。但当前阶段不做——YAGNI。

  业务规则落地

  MOQ强制校验规则：
  1. 步进器初始点击 → qty = moq（跳过1到moq-1的所有值）
  2. 减操作 → 如果当前qty > moq，以moq为步长减少
               如果当前qty = moq且再减 → qty = 0（归零）
  3. 加操作 → 以moq为步长增加（B2B场景：+1箱没有意义）
  4. 点击"加购"时检查 → qty >= moq 才允许加入
  5. 如果qty < moq就点击加购 → Toast提示"最小起订量为X箱"，自动设置为moq

  浮层互斥规则：
  - 有pending配置量 → 显示"一键全部加购"浮层（绿色）
  - 购物车有商品且无pending配置量 → 显示"查看清单"浮层
  - 两者互斥，优先级：配置 > 购物车

  四、商品详情页（ProductDetailView）

  业务场景

  批发商在扫完目录后，对某个商品产生兴趣，需要了解更多信息来做采购决策。这里的关键问题是：批发商做决策需要什么信息？

  和C端不同，C端消费者买一瓶酒关心的是"好不好喝"、"有没有人评价好"。批发商买一批酒关心的是：
  - 价格和折扣（我的利润空间是多少）
  - 规格参数（酒精度、原产地、包装——我的客户（酒吧/餐厅）能不能接受）
  - 库存够不够（我要的量能不能一次满足）
  - 物流信息（整箱重量——运费怎么算）

  甲方原始诉求

  ▎ "批发商点进去要看清楚这个酒什么度数、哪里产的、一箱多重。不能光看个图片。还有库存要显示清楚，别让他下了单我们告诉他
  ▎ 没货。"

  设计决策原因

  为什么详情页的步进器步长是1，和首页的MOQ步长不同？

  首页步长=MOQ是让用户快速达到起订量。但详情页用户已经做出了"我要买这个"的决策，他需要精确控制数量——可能他的客户只需要15
  箱，但MOQ是12箱，他想要的是13、14、15中的某个值，而不是"12"或"24"的倍数。

  这是**从"浏览模式"切换到"采购模式"**的信号变化。所以详情页的步进器步长改为1，同时保留MOQ底线检查（低于MOQ时自动弹回）
  。

  为什么做了分享按钮？

  在B2B场景中，分享是一个高频需求——不是分享到社交媒体（如C端的"分享到朋友圈"），而是分销商把产品信息发到自己的客户群里。
  批发商经常需要把某款新到的酒发给他的下线客户（酒吧、餐厅）看："这个酒新到的，你们要不要？"

  所以这里的"分享"实际上是B端分销链路中的一个环节，不是社交分享。

  为什么做了降价通知按钮？

  批发商对价格敏感。很多批发商会在某个商品上设置"盯住"——等降价了再买。降价通知按钮让批发商可以"标记"一个商品，后续价格变
  动时收到推送。

  为什么库存用进度条可视化？

  数值（"库存：1240件"）对批发商来说信息密度不够——他需要的是 "够不够我买" 的直觉判断。进度条给了这种直觉：
  - 绿色+长条 → 库存充足，随便买
  - 靠近红色区域 → 库存紧张，赶紧下单

  进度条的阈值设计：以2500箱为100%，库存 > 1200为绿色充足区域，<
  300为红色紧张区域。这个阈值基于业务经验：一个批发商一次典型采购量是12-50箱，300箱库存对单一批发商来说已经"挺紧张了"。

  业务规则落地

  步进器规则（详情页）：
  1. 初始值 = moq（不能从0开始选，起订量是底线）
  2. 步长 = 1（精确控制）
  3. 下限保护：如果试图减到 < moq，自动回弹到moq + Toast提示
  4. 输入框：如果手动输入 < moq，自动重置为moq + Toast提示

  库存进度条规则：
  - 满刻度 = 2500（基于最大库存值设定）
  - 颜色：绿色（充足）→ 黄色（一般）→ 红色（紧张）

  五、购物车与结算（CartView + 支付弹窗）

  业务场景

  B2B购物车和C端购物车有本质区别。C端购物车是"放进去再看看、比价、优惠券、犹豫要不要买"。B端购物车是**"采购清单——确认了
  就要执行"**。

  批发商在结算环节关心的核心问题：
  1. 总价对不对（含折扣）
  2. 额度够不够（授信支付）
  3. 流程不可逆（点了确认就是合同成立）

  甲方原始诉求

  ▎ "批发商选好了之后，能看到总价多少、折扣多少、要付多少。然后从我们给他的授信额度里直接扣。不能超额度。点了确认就不能
  ▎ 反悔了。还有，有些人老是不起订量就勾选结算，系统要拦住他们。"

  翻译成需求语言：
  - 购物车要显示完整费用明细（小计 → 折扣 → 应付）
  - 支付用授信额度，不是微信/支付宝
  - 额度不够要阻断，并告知如何充值
  - 点确认支付即代表合同成立
  - MOQ不足的商品不能结算

  设计决策原因

  为什么做了5%大宗协议折扣？

  这是模拟B2B行业中的"年度框架协议折扣"——批发商和工厂在年初签订年度采购框架，约定全年采购量和对应的折扣率。系统中的5%折
  扣代表的不是"促销优惠"，而是已经签订的合同折扣。所以它不是像C端优惠券那样可叠加、可取消的——它是固定的、绑定的。

  为什么支付方式是"授信扣款"而不是微信/支付宝？

  酒水B2B行业的结算惯例是"先货后款"——批发商先提货，月底（或按账期）结款。授信额度就是工厂给批发商的"信用账期"的数字化体
  现。

  - 批发商下单时不需要掏钱（不需要微信余额、银行卡余额）
  - 系统从授信额度中扣除订单金额
  - 批发商在账期内还款后，额度恢复

  这个机制模拟的是真实B2B的信用交易模式，而不是C端的即时交易。

  为什么支付弹窗要设计得这么"重"（授信额度卡+费用明细+法律声明）？

  因为B端支付不是"点击按钮→扣钱"这么简单。每一笔B2B订单都是一份法律合同。支付弹窗的每一个设计元素都是为了让批发商在点击"
  确认"之前，清晰知道自己在做什么：

  - 授信额度卡（深色背景 + 大字金额）→ "这是企业的钱，不是个人的"
  - 费用明细 → "你买的东西每项多少钱，你自己核对过"
  - 法律声明 → "点击确认代表企业授权，有法律效力"
  - 按钮文案"确认，安全授信扣款" → "授信"这个词重复出现，强化B端交易的专业感和严肃性

  如果像C端那样设计一个轻盈的支付弹窗，批发商可能会因为"点得太快"而产生纠纷——"我没注意看就付了，这个订单我不认"。B端设计
  要防止这种事情发生。

  为什么额度不足时的处理方案是"请拨打常州厂财务部充值"而不是在线充值？

  这不是技术做不到——而是业务流程不允许在线充值。B2B的授信额度不是"充100块得100块"——它是基于企业信用评估、年度合同、担保
  等线下的金融流程。额度不足时，批发商需要和工厂财务沟通（可能是追加合同、加快回款、申请临时额度），这些不能靠纯线上完成
  。

  所以阻断文案给了明确的行动路径——"打电话给财务部"，而不是冷冰冰的"额度不足"。

  为什么支付成功后自动跳转到订单页而不是留在购物车？

  这是一个微妙的体验设计。支付成功后，已支付的商品从购物车中清除，购物车变成空或只剩下未勾选的商品。如果留在购物车，用户
  看到的是"空了一部分"的购物车，会困惑。直接跳转到订单页，用户看到的是"新订单已生成"——这是一个正向反馈，告诉用户"你做的
  事情有结果了"。

  业务规则落地

  结算前校验规则：
  1. 未勾选任何商品 → "请至少勾选一个商品"（阻断）
  2. 勾选的商品中有MOQ不足 → "已选商品中存在起订量不足"（阻断，标红）
  3. 应付金额 > 授信额度 → "额度不足，请拨打常州厂财务部充值"（阻断）
  4. 全部通过 → 显示支付弹窗

  支付流程：
  1. 用户点击"确认，安全授信扣款"
  2. 显示加载状态（1.2秒，模拟处理中）
  3. 显示支付成功动画（1.5秒）
  4. 自动跳转到订单页
  5. 授信额度实时扣减
  6. 已支付商品从购物车清除

  折扣规则：
  - 大宗采购协议折扣：固定5%
  - 折扣计算方式：(商品单价 × 数量) × 5%
  - 最终金额保留到分（¥xxx.xx）

  六、订单中心（OrdersView）

  业务场景

  批发商下完单后，最关心的是：货发了没有？到哪了？ 在传统模式中，批发商要问业务员，业务员要问仓库，仓库要问物流——信息层
  层传递，每个环节都有延迟和失真。批发商需要一个自助查单的入口。

  甲方原始诉求

  ▎ "批发商下单后能自己看到订单状态，不用每次打电话来问'我的货发了没有'。订单状态要清楚：下了单是'待处理'，我们发货了是'
  ▎ 发货中'，他们收到了是'已完成'。"

  设计决策原因

  为什么订单状态只有三个（待处理/发货中/已完成）？

  真实B2B订单流程远比这个复杂（待审核 → 已审核 → 拣货中 → 打包中 → 已出库 → 运输中 → 派送中 → 已签收 →
  已完成），但甲方故意简化成三个状态——为什么？

  因为这是面向批发商（买方）的视图。批发商不需要知道仓库内部的拣货、打包、出库流程。他们只需要知道："厂家有没有在处理"→"
  有没有发货"→"有没有到"。三个状态刚好覆盖了这三个关注点。

  而供应端（业务员/仓库）的内部状态复杂度在 SupplierView 中处理，那里有更细的流转控
  制。

  为什么做了"再次购买"功能？

  定期复购是B2B的核心场景之一。很多酒吧/餐厅每周订同样的货——每次都是同样的几款酒，同样的数量。在传统模式中，他们每周要重
  新给业务员发一遍订单。在系统中，"再次购买"让批发商可以直接复用历史订单，不需要重新选一遍商品。

  这个功能实现起来非常简单（把历史订单的商品复制到购物车），但对批发商的效率提升非常显著——尤其是对固定采购的批发商。

  为什么订单卡片不做"取消订单"或"申请退款"？

  这是B2B和C端的又一个重要区别。在B2B交易中，一旦订单生成（授信扣款完成），它就是一个具有合同效力的采购单。批发商不能像
  在淘宝上那样"取消订单"——因为货物可能已经在拣货、打包甚至出库了。

  如果要取消订单，需要走线下的异常流程（联系业务员 → 业务员确认 → 仓库拦截出库 →
  财务处理退款）。这个流程不能简单做成一个按钮。所以订单列表中没有"取消"按钮，只有供应端有异常处理功能。

  业务规则落地

  筛选规则：
  - Tab切换：全部 / 待处理 / 发货中 / 已完成
  - 每个Tab显示对应状态的订单列表

  再次购买规则：
  1. 将历史订单所有商品加回购物车（数量累加）
  2. 如果有商品已在购物车中 → 合并数量
  3. 操作完成后自动跳转到购物车页
  4. 不修改历史订单本身

  订单不可取消（无取消按钮/无退款入口）

  ---
  七、个人中心与企业工作台（ProfileView）

  业务场景

  批发商在使用平台的日常中，除了下单之外还有一系列管理需求。但与C端不同，这些需求都是企业级的——管理的是"公司的事"而不是"
  个人的事"。

  甲方原始诉求

  ▎ "批发商能管理自己的收货地址，有时候送到常州仓库，有时候送到上海门店。还要能看到还有多少额度可以用。如果有问题，能联
  ▎ 系到我们的客服。"

  设计决策原因

  为什么个人中心展示的是企业信息而不是个人信息？

  B端用户的操作是"代表企业"进行的。所以个人中心展示的不是头像、昵称、个性签名，而是：
  - 企业名称（高鑫零售有限公司）
  - 联系人（李远——他是企业授权的采购人）
  - 客户ID（LX-88204-BC——企业对账用的编号）

  这种设计强化了"这是企业行为"的心理认知。如果展示的是李远的个人头像和昵称，批发商可能会觉得"这是我个人的事"，下单时不够
  严肃。

  为什么地址管理需要完整的CRUD（增删改）？

  批发商通常有多个收货地址：可能是不同仓库、不同门店、不同分销点。地址簿的设计考虑了以下场景：
  - 新增：开了一个新门店/新仓库，需要增加收货点
  - 设为默认：大部分货发到主仓库，默认地址减少每次选择的操作
  - 删除：某个门店关闭了，不再需要往那里发货

  这个功能看起来很简单，但在B2B中非常重要——发错地址会导致退换货成本。

  为什么做了消息中心（客服聊天）？

  即使在数字化系统中，仍然有一些事情需要人工沟通——比如额度问题、特殊价格申请、物流异常处理。消息中心的作用是：

  1. 替代微信沟通——所有业务沟通留在平台内，有记录可追溯
  2. 智能自动回复——常见问题（物流、财务）由系统自动应答，减少客服压力
  3. 沟通闭环——批发商在平台内可以完成"问问题→得到答案"的全流程

  自动回复的逻辑：
  - 包含"保税/出库/发货" → 触发物流模板回复
  - 包含"额度/汇款/扣款" → 触发财务模板回复
  - 其他 → 默认自动回复（"您的消息已进入专属客户经理处理通道"）

  这个功能在演示中特别有效——它展示了系统的"智能"能力，而不只是"静态页面"。

  为什么做了三个设置开关？

  三个设置开关对应的都是实际的业务场景：

  - 极速离线数据缓存：仓库和地下室信号差，需要缓存商品图片
  - 价格降价专属推送：批发商盯住的商品降价时收到短信通知
  - 出入库消息同步：订单发货时，提单PDF发送给物流部

  这三个开关不是凑数的——每一个都对应一个真实用户的真实投诉场景。不设开关也行，但给用户"控制权"是好的设计。

  业务规则落地

  地址管理规则：
  1. 新增地址时，如果当前无任何地址 → 自动设置为默认地址
  2. 设置新默认地址 → 取消原默认地址的默认标记
  3. 删除默认地址 → 其他地址不会自动变为默认（需手动设置）
  4. 地址表单校验：收件人、电话、详情为必填

  消息中心规则：
  1. 用户发送消息后，1秒触发自动回复
  2. 自动回复基于关键词匹配（关键词检测在App.tsx中）
  3. 消息气泡区分颜色：用户（深色）vs 客服（浅色）

  八、供应端工作台（SupplierView）

  整体设计逻辑

  供应端是整个平台最复杂的模块（约1400行代码），因为它需要容纳三种完全不同角色的工作台。

  甲方原始诉求

  ▎ "我们厂里的人也要用这个系统。业务员要在上面处理批发商的订单、安排发货；仓库要核单出库；我要能在后台配置价格、加商品
  ▎ 、看数据。但是这三个人看的东西完全不一样，不要搞一个菜单让他们自己找。"

  核心设计决策原因

  为什么供应端不做成"后台管理菜单"的形式？

  传统做法是一个后台管理页面，左边菜单栏有十几个菜单项：订单管理、商品管理、客户管理、仓库管理、物流管理、数据报表……每个
  人都在同一个界面操作，靠权限控制哪些菜单可见。

  但在移动端上，这种做法非常糟糕——因为屏幕小，菜单本身就占用了大量空间。更重要的是，三个角色的工作流完全不同，给他们一个
  统一的"后台"让他们自己找功能，效率极低。

  所以设计上采用 "角色即视图"——供应端不是一个"后台"，而是三个独立的"工作台"，每个工作台只为该角色的核心任务优化。

  ---
  8.1 业务员视图

  为什么业务员的子Tab是"合同订单簿 / 异常协控单 / 大宗采购商"？

  这三个Tab对应业务员的三个核心工作流：

  1. 合同订单簿（核心）：查看所有订单，审核通过，分配物流。业务员每天上班第一件事就是打开这个Tab看有没有新订单。
  2. 异常协控单（例外处理）：当订单因为各种原因（天气、库存、物流）不能正常发货时，集中处理。单独拎出来做一个Tab而不是混
  在订单列表中，是因为异常订单需要特殊关注和集中处理——混在正常订单里很容易被忽略。
  3. 大宗采购商（客户管理）：查看客户信息和历史采购数据。业务员需要了解每个客户的采购习惯和信用情况，以便做客户维护。

  为什么物流商选项有"顺丰B2B专列"和"常州保税专配车队"？

  这两个选项是有业务背景的："顺丰B2B专列"对应外省/跨区域配送，"常州保税专配车队"对应常州本地配送。不同的物流商对应不同的
  成本和时效，业务员需要根据订单目的地和客户要求选择合适的物流方式。

  ---
  8.2 仓库核单员视图

  为什么做了扫码核单模拟器而不是直接做一个"一键出库"按钮？

  如果只是为了"功能上有"，直接做一个"一键出库"按钮是最简单的——点击 → 状态变成"发货中" → 完事。

  但扫码核单模拟器的设计反映了一个深层需求：仓库物理核对这个环节是不可跳过的。 在传统流程中，核单员需要到货架上找到对应
  商品，核对数量，确认无误后才能出库。跳过这个环节直接出库，拿错货、数错数的风险很高。

  扫码核单模拟器把线下物理流程映射到数字界面：
  1. 选择待核订单（从货架上找到对应的订单）
  2. 逐项扫码（用扫码枪扫描每个SKU的条码）
  3. 确认所有SKU已扫（确认所有商品都齐备了）
  4. 录入车牌号（装车）
  5. 一键出库（发车）

  每一步都对应一个线下物理动作。虽然当前版本是"点击按钮模拟扫码"，但接口设计已经为真实扫码枪对接预留了替换空间。

  为什么出库时需要录入配送车辆信息？

  这是实际物流管理的需要。出库不只是"东西发出去了"——还需要知道"谁运的、车牌多少、运单号多少"。这些信息后续用于物流追踪和
  签收确认。

  ---
  8.3 管理员视图

  为什么管理员的四个子Tab是"商品管理 / 价格微调 / 供销规则 / 经营数据"？

  这四个Tab对应工厂管理者的四个核心管理职责：

  1. 商品管理（catalog）：增删改商品。工厂要经常上新品、下架老品、调整库存。
  2. 价格微调（pricing）：批量调价。B2B的价格变动比C端频繁——原材料涨价、促销活动、客户级别不同，都需要调价。
  3. 供销规则（rules）：全局规则配置。MOQ倍率、折扣率、授信额度默认值——这些是控制整个平台商业逻辑的参数。
  4. 经营数据（metrics）：数据看板。管理者需要实时了解GMV、订单完成率、库存情况。

  为什么经营数据用SVG柱状图展示各单品库存？

  这个设计有点"大材小用"——在移动端用SVG画柱状图，代码量不小。但它的存在是有理由的：在演示场景中，一个可视化的数据看板比
  一堆数字更有说服力。
  当客户看到彩色的柱状图展示库存数据时，感知是"这是一个正规模的生产系统"，而不是"一个简单的下单页面"。

  个项目的每一个功能细节都不是"拍脑袋"做的，而是从真实的B2B酒水批发业务场景中推导出来的。
